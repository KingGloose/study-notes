1、LSP（语言服务器协议）的工作原理

LSP 是 Microsoft 提出的开放协议，将编辑器与语言工具解耦。语言服务器（如tsserver）运行在独立进程中，编辑器/工具通过 JSON-RPC 与它通信，请求代码补全、跳转定义、错误诊断等功能。 编辑器 /Claude <--JSON-RPC--> Language Server (tsserver) 发送请求 分析 AST，返回结果

💡 这种设计让一个语言服务器可以被 VS Code、Vim、Claude Code 等任意工具复用，无需每个编辑器都自己实现 TypeScript 分析逻辑。

TypeScript 使用结构化类型——只要两个类型的"形状"相同，就认为它们兼容，不管名字是否一样：

  interface Cat { name: string; meow(): void }
  interface Dog { name: string; meow(): void }

  const cat: Cat = { name: "Tom", meow: () => {} }
  const dog: Dog = cat  // ✅ TypeScript 允许！形状相同即兼容

  // 而 Java/C# 是名义类型 —— 不同名字就是不同类型，即使结构相同也不兼容

  ▎ 💡 这正是 LSP 能做语义分析的基础之一——它需要理解 TypeScript 的类型系统，而不只是看名字匹配。
